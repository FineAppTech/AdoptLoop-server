# 작업 워크플로우

> AdoptLoop 구현 작업의 표준 절차. 작업자(사람·AI)가 명시적으로 참조하는 단일 기준.
> 큰 그림(스펙·계획)은 [PROJECT_CONTEXT](PROJECT_CONTEXT.md) → SPEC/PLAN 참조.

## 기본 단위

각 작업 단위 = **TDD 한 사이클** 또는 **한 결정 단위**. PLAN의 Task/Step을 순서대로 따른다.

## 7단계 절차

1. **작업 보고** — 무엇을 할지 먼저 보고 (PLAN의 해당 Task 정의를 읽고).
2. **작업 수행** — PLAN을 직접 참고해 구현. (실행용 보조 스킬은 쓰지 않음 — 아래 *실행 스타일*.)
3. **셀프 검증 + 코드 리뷰** — 사용자에게 보고하기 **전에** 직접 수행.
   - 검증: 가능한 빌드/컴파일/테스트 실행, Expected 대비 실제 확인.
   - 리뷰: 정확성 / PLAN 이탈 / 스타일·일관성 / 단순성 / 잔여물(미사용 import 등).
   - **로직 작업**(엔티티·리포지토리·서비스·컨트롤러·비즈니스/검증/필터 로직, TDD)이면 `/code-review`(low~medium)를 **자동 실행**.
   - **트리비얼 작업**(빌드/설정/문서/단순 기계적 변경)은 인라인 셀프 리뷰만.
   - 무거운 `/code-review ultra`(클라우드·과금)는 자동 실행하지 않음 — 필요 시 사용자가 직접 트리거.
4. **결과 보고 (채팅)** — 검증·리뷰 결과 포함.
5. **보고서 파일 작성** — `docs/reports/m{N}-task-{X.Y}-{slug}.md`. 아래 *보고서 규칙*.
6. **사용자 승인** — 명시적 "승인"(또는 동등한 커밋 go-ahead)을 받아야만 7로 진행.
   - ⚠️ 커밋 세부(위치/분리 방식 등)를 묻는 질문에 답을 받은 것은 **커밋 승인이 아님**. 별도로 "커밋할까요?"를 묻고 기다린다.
7. **커밋** — 6의 명시 승인 후에만. 보고서 파일 포함.

## 보고서 규칙

- 위치: `docs/reports/`, Task별 파일 1개 (Milestone 종합 아님).
- 필수 항목: **변경 파일 목록** / **검증·리뷰 결과** / **결정·이탈 사항**.
- 마크다운 표(`| ... |`) 앞에는 **반드시 빈 줄**을 둔다 (없으면 렌더러가 표로 인식 못 함).

## 커밋 규칙

- **태스크별 단일 커밋** — 한 Task의 작업을 여러 커밋으로 쪼개지 않는다.
- 명시 승인 후에만 커밋 (6단계).
- `push` / PR 등 외부에 보이는 액션은 사용자 확인 후.
- 원격 `main`은 보호 — 모든 작업은 feature 브랜치 + PR. Milestone 단위로 PR 분리.

## 네이밍 규칙

- **약어 금지** — 패키지/디렉토리/식별자는 전체 단어로. 예) `repo` ❌ → `repository` ✅.
- DB: snake_case 소문자, FK 컬럼은 `단수테이블명_id`.

## 실행 스타일

- 브레인스토밍·계획 수립까지는 보조 스킬을 써도 좋으나, **실행 단계는 PLAN 문서를 직접 가이드 삼아 수행**한다 (실행 오케스트레이션 스킬 미사용).
- 진행 상황은 텍스트로 보고하며, PLAN 파일의 체크박스를 일일이 마킹하지 않는다(커밋 노이즈 방지).

## 의사결정 기록 (ADR)

- 구현 중 SPEC/TECH_STACK을 바꿔야 하는 결정은 편집 대신 ADR 추가(`Supersedes: ADR-XXXX`).
- 옛 ADR은 수정/삭제하지 않고 status만 `Superseded by ADR-YYYY`로 갱신.
- 일반 구현 디테일(메서드명, 의존성 좌표, 테스트 도구 선택 등)은 ADR 대상이 아니다.
- 테스트 강제: 컨트롤러 endpoint 테스트는 REST Docs `documentApi()` 호출 필수 → [ADR-0009](adr/0009-spring-restdocs-enforcement.md).
